The problem with static lead lists

Most outbound tooling — Apollo, Clay's default enrichment, a purchased list — answers "who fits the ICP profile." It never answers "who is doing something right now that makes this the moment to reach out." A profile match from six months ago is stale by the time an SDR sends the email.

The fix isn't a bigger list. It's scoring real, current activity and only acting on genuine signal — the same principle whether the domain is Amazon sellers or, as shown here, open-source companies.

The result — real GitHub data, pulled live

Composite score = trailing-4-week commit velocity vs. the 4 weeks before it. Deceleration is scored 0, not negative — a slowdown isn't a buying signal, and the honest answer to "no signal" is 0, not a fabricated penalty.

What this isn't

This score alone doesn't prove intent — it's one signal, not a verdict. The point of the mechanism isn't "this one number replaces judgment," it's that a real, current, checkable signal is a far better trigger than a static list that doesn't know anything changed.

Closing the loop — signal to draft

Scoring the signal is only half of "signal-based intent orchestration." The other half: use it to draft something real. draft_outreach.py pulls a few more real, current facts about the top-scoring company and has Claude draft a short message grounded only in those facts — never a generic pitch.

The actual scoring code

Full script: signal_score.py — runnable, no API key required.

# composite score: reward genuine acceleration only
# deceleration -> 0, not a fabricated negative signal
def signal_score(velocity_change_pct):
    if velocity_change_pct is None:
        return 0
    return max(0, round(velocity_change_pct, 1))

# velocity: trailing 4wk commits vs. the 4wk before that,
# computed from GitHub's public /stats/commit_activity endpoint
last4 = sum(w["total"] for w in activity[-4:])
prev4 = sum(w["total"] for w in activity[-8:-4])
change_pct = (last4 - prev4) / prev4 * 100
